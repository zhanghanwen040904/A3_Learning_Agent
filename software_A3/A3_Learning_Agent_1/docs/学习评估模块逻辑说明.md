# 学习评估模块逻辑说明

本文档基于当前项目代码实际实现整理，目标是把“学习评估”模块的真实运行逻辑讲清楚，包括：

- 评估模块整体链路
- 知识库、题库如何构建
- 个性化出题如何完成
- 学生答案如何评分
- 评分后如何更新学习画像
- 相关数据库表如何存储
- 当前实现特点与限制

本文档对应的核心代码位置：

- `backend/api/evaluation_api.py`
- `backend/ai/assessment_pipeline.py`
- `backend/ai/structured_course_data.py`
- `backend/ai/agents/evaluator_agent.py`
- `frontend/src/views/EvaluationView.vue`
- `frontend/src/api/index.js`
- `backend/db/schema.py`

---

## 1. 模块整体目标

当前“学习评估”模块的目标不是单纯出几道题，而是形成一条完整闭环：

1. 从课程语义化知识数据中构建知识点库和题库。
2. 根据学生画像和历史薄弱点，生成个性化题目。
3. 根据参考答案、关键词和覆盖度，对学生答案进行规则化评分。
4. 将成绩、薄弱原因、知识点掌握度写回数据库。
5. 进一步刷新右侧学习画像、薄弱知识点和下一步学习建议。

所以它本质上是一条“题库驱动 + 规则评分 + 画像回写”的评估流水线。

---

## 2. 当前模块是否调用大模型

当前“学习评估”部分，核心判题逻辑 **不是直接调用大模型 API 评分**，而是使用了本地代码规则进行判分。

这一点非常重要。

虽然 `EvaluatorAgent` 挂了 `spark_chat` 工具描述，但实际 `grade()` 方法里当前走的是：

- 文本归一化
- 关键词命中
- 参考答案覆盖率
- 字符重合度
- 规则映射成分数

也就是说，当前评估模块是：

- 出题：基于知识语义数据自动生成
- 评分：基于规则
- 画像更新：基于评分结果写回数据库

这样做的优点是稳定、可控、无额外 API 成本；缺点是语义理解深度不如真正的 LLM 判题。

---

## 3. 模块整体流程

整体流程可以概括为：

```text
语义化课程知识数据
    ↓
构建 knowledge_points.json 与 question_bank.json
    ↓
读取学生画像 + 历史掌握记录
    ↓
个性化选题
    ↓
前端展示题目
    ↓
学生提交答案
    ↓
规则评分
    ↓
写入 quiz_result / mastery_record / learning_event
    ↓
刷新 student_profile 中的 weak_points / course_progress
    ↓
前端 summary 重新展示画像、薄弱点、下一步任务
```

## 4. 知识库和题库是怎么建的

### 4.1 数据源不是直接 PDF，而是语义切片后的 JSON

当前评估模块使用的底层数据不是直接读取 PDF 文本，而是读取：

- `rag_data/raw_extracted/software_engineering/semantic_json/*.json`

这些文件是已经做过语义切片后的课程知识数据，每个 JSON 文件中包含多个 `knowledge_chunks`。

在代码中，入口是：

- `backend/ai/structured_course_data.py`
- 函数：`load_semantic_chunks()`

### 4.2 `structured_course_data.py` 做了什么

这个模块负责把语义切片数据清洗成适合评估模块使用的标准知识块。主要步骤如下：

#### （1）定位语义数据目录

通过：

- `generated_kb_dir()`
- `semantic_json_dir()`

最终定位到：

- `rag_data/raw_extracted/software_engineering/semantic_json`

#### （2）清洗文本

使用以下函数做预处理：

- `clean_text()`
- `normalize_title()`
- `normalize_line()`
- `preprocess_text()`
- `split_sentences()`
- `summarize_text()`

清洗目标包括：

- 去掉多余空格和连续空行
- 去掉页码、题头、噪声行
- 去掉类似“来源文件”“知识路径”“起始页”等非正文信息
- 过滤太短、无意义、像编号残片一样的标题

#### （3）过滤无效知识块

通过：

- `is_useful_title()`
- `is_useful_chunk()`

过滤掉以下内容：

- 标题过短
- 纯编号式标题
- 内容长度过短
- 噪声占比过高
- 无实际教学价值的块

#### （4）统一输出标准知识块

每个知识块最终整理成如下结构：

- `chunk_id`
- `source`
- `course`
- `knowledge_point`
- `section_title`
- `section_path`
- `section_path_text`
- `pages`
- `knowledge_type`
- `tags`
- `content`
- `summary`

也就是说，评估模块后面使用的不是杂乱的原始文本，而是已经结构化的“知识块”。

## 5. 题库与知识点树如何生成

核心代码在：

- `backend/ai/assessment_pipeline.py`

### 5.1 输出文件

评估模块最终生成两个核心文件：

- `rag_data/assessment/knowledge_points.json`
- `rag_data/assessment/question_bank.json`

其中：

- `knowledge_points.json`：知识点树/知识点节点集合
- `question_bank.json`：题库

### 5.2 当前真实规模

根据当前项目文件实际统计：

- 知识点数：`322`
- 题目数：`488`

题型分布：

- `short_answer`：488 道

难度分布：

- `basic`：462 道
- `improve`：26 道

所以当前系统的题型还比较集中，主要是问答型简答题。

### 5.3 知识点树构建方式

在 `_build_from_semantic_chunks()` 里，系统会遍历每个语义知识块的：

- `section_path`

然后逐层建立知识点节点。

例如一个知识块的路径可能类似：

- 软件工程 / 软件工程学概述（下） / 软件生命周期

系统会逐层建立：

1. 软件工程
2. 软件工程 / 软件工程学概述（下）
3. 软件工程 / 软件工程学概述（下） / 软件生命周期

每个知识点节点包含：

- `id`
- `name`
- `path`
- `parent_id`
- `document`
- `summary`
- `keywords`
- `children`

这样前端就能拿到一棵可展示、可追踪的知识树。

### 5.4 题库生成方式

题库也是在 `_build_from_semantic_chunks()` 中构造的。

每个有效知识块基本会生成一题，系统会为它自动生成：

- `id`
- `knowledge_point`
- `knowledge_path`
- `source_document`
- `question_type`
- `difficulty`
- `prompt`
- `reference_answer`
- `explanation`
- `common_mistake`
- `scoring_points`
- `keywords`
- `pages`
- `tags`
- `knowledge_type`
- `chunk_id`

同时，为避免重复，会做一次去重：

- 去重键：`(knowledge_path, prompt)`

如果同一路径生成出完全相同的题干，则跳过后续重复题。

## 6. 一道题是怎么自动生成出来的

题库中的每道题，不是手写录入的，而是由知识块自动派生出来。

### 6.1 题干生成：`_make_prompt()`

题干生成依据：

- `knowledge_point`
- `knowledge_type`
- `tags`

当前规则如下：

- 如果是 `comparison`，题目更倾向“解释区别”
- 如果是 `method / process_model / phase`，题目更倾向“按步骤、阶段解释”
- 如果有标签，则提示“围绕该知识点并适当提及若干标签”
- 默认情况：直接要求解释该知识点

因此当前题目生成属于“模板化问答题生成”，不是自由生成式命题。

### 6.2 参考答案生成：`_make_reference_answer()`

参考答案来源于知识块正文 `content`。

当前做法：

- 使用 `summarize_text(content, max_sentences=2)`
- 即从清洗后的知识块中抽取前 2 个有效句子作为参考答案

所以参考答案本质上是：

- 从原始课程知识内容提炼出的短摘要

### 6.3 解析生成：`_make_explanation()`

解析不是简单复制参考答案，而是会做一定的结构化拼接。

它会：

1. 先强调“这题考查的是不是背结论，而是是否真正理解这个知识点”
2. 抽取正文前几句，拼成：
   - 核心意思
   - 进一步展开方向
   - 如果想得高分可以补充哪些内容

因此当前解析是“规则化生成的讲解文本”，比参考答案长，更偏教学化。

### 6.4 常见错误生成：`_make_common_mistake()`

系统会根据知识块文本特征判断：

- 是不是比较题
- 是不是流程题
- 是不是优缺点题
- 是不是应用场景题

然后生成对应的“易错点/常见失分点”。

例如：

- 比较题：容易只解释单个概念，没有区分相近概念
- 流程题：容易只列关键词，不按顺序说明
- 应用题：容易只背定义，说不出适用场景

如果无法匹配到特定模式，就退化为通用提示。

### 6.5 得分点生成：`_make_scoring_points()`

系统会自动列出 3~4 条得分点，来源包括：

- 该知识点本身的定义
- 从正文提取的关键词
- 如果内容中出现“应用/场景/案例”，补一个应用类得分点
- 如果内容中出现“步骤/流程/阶段”，补一个顺序表达类得分点

所以得分点不是人工标注，而是根据知识块自动生成。

## 7. 个性化出题逻辑

核心函数：

- `generate_personalized_questions(profile, mastery_records, count, knowledge_point)`

### 7.1 输入来源

出题时会综合两类用户信息：

#### （1）学生画像 `student_profile`

重点用到：

- `weak_points`
- `study_goal`
- `course_progress`

#### （2）掌握记录 `mastery_record`

重点用到：

- `knowledge_point`
- `mastery_score`

### 7.2 如何识别“关注知识点”

函数 `_parse_profile_focus()` 会把下面两部分合并：

1. 画像里的 `weak_points`
2. 历史掌握度 `< 75` 的知识点

得到一个 `focus_points` 列表。

如果前端还传了指定知识点 `knowledge_point`，则会优先加入这个列表最前面。

### 7.3 如何给题目打分

系统会遍历题库中每一道题，把以下字段拼成一个搜索串：

- `knowledge_point`
- `knowledge_path`
- `prompt`
- `keywords`
- `tags`

然后按以下规则累积分：

- 每命中一个关注知识点，加 4 分
- 如果是前端显式指定的知识点，再额外加 6 分
- 如果题目难度是 `basic`，再加 1 分

然后按分数从高到低排序。

### 7.4 如何选题

选题规则如下：

1. 优先选得分高的题
2. 尽量避免同一 `knowledge_path` 重复过多
3. 如果有关注点但某题完全不命中，则会被跳过
4. 如果题还不够，就从整个题库补足

最终返回：

- `focus_points`
- `recommended_knowledge_points`
- `knowledge_tree`
- `questions`
- `question_count`

所以当前“个性化”主要体现在：

- 优先围绕画像薄弱点出题
- 支持指定某个知识点强制出题
- 偏向先出基础题

## 8. 学生答案是怎么评分的

核心代码：

- `backend/ai/agents/evaluator_agent.py`
- 核心方法：`grade()`

### 8.1 评分输入

评分时输入的内容包括：

- `question`
- `answer`
- `reference_answer`
- `knowledge_point`
- `explanation`
- `common_mistake`
- `scoring_points`

也就是说，评分并不是只看“学生答案 vs 参考答案”，而是会综合该题的教学辅助信息。

### 8.2 文本预处理

评分前先做归一化：

- 去空格
- 去标点
- 去换行差异
- 统一大小写

对应函数：

- `_normalize_text()`

目的是避免因为格式不同导致判分偏差。

### 8.3 关键词提取

通过 `_extract_keywords()`，评分器会从以下来源提取关键词：

1. `knowledge_point` 本身
2. 预设领域词表 `DOMAIN_KEYWORDS`
3. `scoring_points` 中可提炼的关键词

最终得到最多 6 个关键词。

### 8.4 三个核心评分指标

评分主要使用三个指标：

#### （1）关键词命中率 `keyword_ratio`

学生答案中出现了多少个核心关键词。

#### （2）语义单元覆盖率 `unit_ratio`

把参考答案切成若干“语义单元”后，检查学生答案覆盖了多少单元。

对应函数：

- `_split_units()`
- `_unit_coverage_ratio()`

#### （3）字符重合率 `char_ratio`

看学生答案和参考答案字符集合的重合程度。

对应函数：

- `_char_overlap_ratio()`

### 8.5 实际判分规则

当前规则大致如下：

#### 情况 A：空答案

- 分数 = 0

#### 情况 B：与参考答案完全一致

- 分数 = 100

#### 情况 C：参考答案被完整包含在学生答案中

- 分数 = 95

#### 情况 D：有关键词可用时

计算：

- `blended_ratio = max(keyword_ratio, unit_ratio * 0.75 + char_ratio * 0.25)`

然后映射到分数：

- `>= 0.9` → 92 分
- `>= 0.75` → 84 分
- `>= 0.6` → 74 分
- `>= 0.4` → 60 分
- `< 0.4` → 40 分

#### 情况 E：没有关键词时

- 答案长度 ≥ 20：75 分
- 否则：55 分

### 8.6 正误判定

当前规则中：

- `score >= 70` 视为 `is_correct = true`

这意味着当前模块并不是严格二值“对/错”，而是：

- 70 分以上算基本掌握
- 70 分以下算仍需巩固

### 8.7 反馈文本怎么来

根据分数段，系统返回不同反馈：

- 90+：回答非常完整
- 75+：回答基本正确
- 60+：部分正确但缺少关键点
- 60 以下：回答不够充分，建议回到知识点原文重新梳理

同时还会返回：

- `matched_keywords`
- `missed_keywords`
- `weak_reason`

其中 `weak_reason` 会优先用“缺失的关键点”来描述学生薄弱原因。

## 9. 评分后如何更新学习画像

评分不是只返回前端，后端还会立刻把结果写回数据库并刷新画像。

核心在：

- `backend/api/evaluation_api.py`

### 9.1 写入 `quiz_result`

每次提交题目后，都会保存一条答题记录，包括：

- 用户 ID
- 题目
- 学生答案
- 参考答案
- 分数
- 反馈
- 知识点

### 9.2 更新 `mastery_record`

通过 `_upsert_mastery()`，对当前知识点进行 upsert：

- 若该用户此前没做过这个知识点，就插入
- 若已经有记录，就更新分数和薄弱原因

注意：当前是“一个用户 + 一个知识点”只保留一条掌握记录，最新评分会覆盖旧评分。

### 9.3 写入 `learning_event`

提交题目后，还会额外写一条学习事件：

- `event_type = finish_quiz`
- `knowledge_point`
- `detail` 中记录：
  - `score`
  - `question`
  - `is_correct`
  - `missed_keywords`

这为后续做学习轨迹分析保留了数据基础。

### 9.4 刷新 `student_profile`

函数 `_refresh_profile_after_evaluation()` 会重新读取全部 `mastery_record`，然后：

- `mastery_score < 70` 的归为薄弱项
- `mastery_score >= 85` 的归为优势项

并回写到 `student_profile`：

- `weak_points`
- `course_progress`

更新逻辑是：

#### 若存在薄弱项

将前 5 个薄弱知识点拼成 `weak_points`

#### 若存在优势项

将前 5 个优势点和薄弱点一起组成 `course_progress`

#### 若只有薄弱项、没有优势项

则 `course_progress` 会变成一段“当前薄弱点 + 建议优先复习”的描述

因此，右侧学习画像并不是静态写死的，而是由评测结果驱动更新的。

## 10. 前端评测页是如何工作的

核心文件：

- `frontend/src/views/EvaluationView.vue`

### 10.1 页面功能

当前前端页面提供了几个主要能力：

1. 查看评估 summary
2. 重建题库
3. 根据画像生成题目
4. 单题模式 / 批量模式作答
5. 查看评分、解析、易错点、得分点、遗漏关键词
6. 实时查看右侧学习画像、最近得分、薄弱点、下一步任务

### 10.2 前端调用的接口

接口封装在：

- `frontend/src/api/index.js`

评估相关接口包括：

- `evaluationApi.bankStatus()`
- `evaluationApi.rebuildBank()`
- `evaluationApi.questions()`
- `evaluationApi.submit()`
- `evaluationApi.summary()`
- `evaluationApi.event()`

### 10.3 页面初始化逻辑

页面 `onMounted()` 时会自动执行：

1. `loadSummary()`
2. `generateQuestions()`

也就是页面一打开就会：

- 拉取已有学习评估情况
- 根据当前画像自动生成一批题

### 10.4 答题后页面如何联动

提交某题后：

1. 前端调用 `/evaluation/submit`
2. 后端返回本题判分结果
3. 当前题卡展示：
   - 得分
   - 反馈
   - 参考答案
   - 解析
   - 易错点
   - 得分点
   - 遗漏关键词
4. 前端再调用 `loadSummary()`
5. 右侧学习画像同步刷新

因此页面上的“画像摘要”“薄弱点”“最近得分”是与评测联动的。

## 11. 评估模块相关数据库表

在 `backend/db/schema.py` 中，学习评估直接相关的表主要有三张。

### 11.1 `quiz_result`

作用：保存每一次具体答题结果。

关键字段：

- `user_id`
- `question`
- `answer`
- `reference_answer`
- `score`
- `feedback`
- `knowledge_point`
- `create_time`

### 11.2 `mastery_record`

作用：保存每个用户在每个知识点上的当前掌握度。

关键字段：

- `user_id`
- `knowledge_point`
- `mastery_score`
- `weak_reason`
- `update_time`

约束：

- `UNIQUE KEY uk_mastery_user_point (user_id, knowledge_point)`

说明当前粒度是：

- 一人一知识点一条记录

### 11.3 `learning_event`

作用：保存学习过程事件流。

关键字段：

- `user_id`
- `event_type`
- `knowledge_point`
- `detail`
- `create_time`

评测场景下最主要的事件是：

- `finish_quiz`

## 12. 当前评估逻辑的“根据什么评估”

如果用最直接的话概括，当前系统评估学生答案主要依据 4 类信息：

### 12.1 题目的参考答案

来源：知识块摘要。

作用：作为学生答案覆盖程度的主要对照基准。

### 12.2 题目的关键词 / 得分点

来源：

- 知识点本身
- 领域关键词词表
- 自动生成的 scoring points

作用：判断学生是否提到了关键知识点。

### 12.3 学生答案与参考答案的覆盖关系

通过：

- 语义单元覆盖率
- 字符重合率

作用：判断学生不是只写了一两个词，而是有没有真正覆盖核心内容。

### 12.4 学生历史画像

作用不直接影响单题判分，但影响：

- 出什么题
- 哪些知识点被重点考查
- 哪些知识点被标为薄弱项

所以“根据什么评估”可以拆成两层：

- 判题层：根据参考答案、关键词、覆盖率
- 选题层：根据画像和历史掌握度

---

## 13. 当前实现的优点

### 13.1 链路完整

不是单独做了一个判题按钮，而是做到了：

- 题库生成
- 个性化选题
- 判题
- 落库
- 画像回写

### 13.2 可解释性强

相比完全黑盒的大模型评分，当前规则评分会明确给出：

- 参考答案
- 解析
- 易错点
- 得分点
- 遗漏关键词

比较适合比赛答辩时展示“为什么这么判”。

### 13.3 与画像联动明显

评估不是孤立模块，而是能够反向更新：

- `weak_points`
- `course_progress`
- 实时学习侧栏

这和赛题要求中的“个性化学习支持”是能对上的。

---

## 14. 当前实现的局限

### 14.1 评分仍是规则评分，不是语义级 LLM 判题

这意味着：

- 对标准答案型、知识点解释型题目效果较稳定
- 对高度自由表达、长篇论述、同义改写丰富的答案，可能不够灵活

### 14.2 题型较单一

当前题库基本都是：

- `short_answer`

缺少：

- 选择题
- 判断题
- 填空题
- 案例分析题
- 编程题/实操题

### 14.3 掌握度更新较“平面”

当前 `mastery_record` 是：

- 一次新成绩直接覆盖该知识点旧成绩

还没有加入：

- 时间衰减
- 多次作答加权平均
- 最近表现优先但保留历史趋势

### 14.4 个性化策略还偏轻量

目前个性化出题主要依赖：

- `weak_points`
- `mastery_score`
- 指定知识点

还没引入更复杂的因素，例如：

- 学习目标优先级
- 专业方向
- 当前课程阶段
- 答题耗时
- 连续错题模式

---

## 15. 结论

当前项目里的“学习评估”模块，已经形成了一个较完整、可演示、可解释的闭环系统：

- 底层基于语义化知识块自动构建知识点树和题库
- 当前真实规模为 322 个知识点、488 道题
- 出题时结合学生画像和掌握记录进行个性化推荐
- 评分时主要依据参考答案、关键词、语义单元覆盖率和字符重合率
- 判题后会把结果写回数据库，并同步更新薄弱点和课程进度画像
- 前端能实时展示评分、解析、得分点、遗漏关键词和学习画像变化

如果从比赛表述上总结，可以把它定义为：

> 一个“基于课程语义知识库 + 个性化选题 + 规则可解释评分 + 画像动态回写”的学习评估子系统。
